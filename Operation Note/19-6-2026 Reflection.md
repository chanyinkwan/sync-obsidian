### 在 McKinsey 流程上,你斷在哪一步

《The McKinsey Mind》把問題解決拆成五步:
1. Framing the Problem(定義問題 + 初始假設 + MECE issue tree)
2. Designing the Analysis(要做哪些分析才能證明它;find the key drivers、don't boil the ocean)
3. Gathering the Data(蒐集數據)
4. Interpreting the Results(每個結果都問 "What's the so what?" + sanity check)
5. Presenting Your Ideas(呈現)

**你把這五步壓縮成只剩第 3 步(蒐集+填)。** 你的 Framing(第 1 步)其實做得好。真正被跳過的是**第 2 步(設計分析)**與**第 4 步(詮釋 / so-what / 合理性檢查)**——而 Ziyi 的四個問題,全部落在這兩步裡。

### 解法:可複製的「送出前閘門」(對抗 ADHD、通用、不綁個案)

Gemini 給的三站(乾淨 / 加權 / 趨勢)方向對,但它是從 Ziyi 這次三個問題反推出來的——換一份財報、換一個 KPI,坑就不一樣,沒法複製。把它升一層,變成不綁個案的通用閘門。
規則只有一條:**任何數字進入「正式交付」之前,先過一次閘門。表還在草稿,手別碰正式文件。**

```
拿到數字  →  先別填正式文件
              │
   ┌──────────┴─ 閘門(每個數字都要過)
   │
   1. 這數字「乾淨」嗎?   → 有沒有外力(併購 / 促銷 / 一次性項目)?
   2. 這數字「怎麼算的」? → 口徑 / 分母 / 加權?是 group 還是我真正要的對象?
   3. 這數字「往哪走」?   → 至少三點才叫趨勢;兩點只是快照
   4. 「So what?」         → 對決策者意味什麼?寫不出一句洞察 = 還沒做完
              │
   全部過  →  才填進正式交付
```

前三問是第 4 步(Interpreting)的具體化;第 4 問就是書裡的 "What's the so what?" 與 sanity check。它不挑個案——任何 KPI 表都能跑同一套。

兩個對 ADHD 特別關鍵的操作原則:
- **外部化,別靠腦記。** 把這個閘門做成螢幕旁便簽或一個 callout。書裡的 issue tree 就是同一招:讓結構替你記住「要問什麼」,大腦只負責回答,不負責「記得要問」。
- **動手前先加一步 Design。** 拿到任務先寫一行:「要回答『這客戶表現如何』,最該看的 3 個關鍵驅動是什麼?」(find the key drivers)先選戰場,再蒐集——這一步能斷掉你「看到空格就想填滿」的反射。

> 一句話總結:你不是資料能力不夠,是把「五步問題解決」做成了「一步填表」。補回第 2 步(先設計要看什麼)與第 4 步(每個數字都過 So-what 閘門),這個失誤就不會再來。
